# 失败实验与调优风险记录

## 1. 动态阈值扫描

本地模型默认使用 0.5 作为判定阈值。分析混淆矩阵后可以看到模型偏保守，存在一定数量的 false negative。我们曾做过阈值扫描，用于观察模型对阈值的敏感性。

需要强调的是：如果直接在 `val.csv` 上选择最佳阈值，会形成验证集调参风险。因此该扫描只作为诊断记录，不作为最终阈值选择依据。正式实验中，我们使用 `train_clean.csv` 内部 dev 切分选择阈值，发现 dev 最佳阈值为 0.51，但应用到 `val.csv` 后 Accuracy 从 0.8803 降到 0.8753，因此最终保留 0.5。

## 2. Event 特征融合尝试

考虑到谣言通常具有事件聚集性，我们曾尝试把数据集中的 `event` 字段作为 One-Hot 特征拼接到 TF-IDF 特征后。但该字段在真实提交场景中未必总是可用，且容易让模型记住事件分布而不是文本证据。

实验中，加入 event 特征后 `val.csv` Accuracy 从 0.8803 降到约 0.8753，没有带来稳定收益。因此最终模型不使用 event 字段，只在分析阶段按 event 统计错误分布。

## 3. L1 正则化降噪尝试

由于词级和字符级 TF-IDF 会产生大量高维稀疏特征，我们尝试过使用 L1 正则化压缩无用特征。实际实验中，L1 正则化需要特定求解器，在当前高维稀疏特征下训练时间更长，并出现收敛不稳定的问题。

考虑到大作业要求代码可运行、结果可复现，最终保留训练稳定、解释清晰的 L2 正则化逻辑回归方案。

## 4. 关于手工话术规则

匿名信源、指控抹黑、暗示隐瞒、极端化表述等模式确实可以作为解释层的证据归纳。例如 `anonymous + obtained` 可以解释为“匿名爆料类线索”，`smear + campaign` 可以解释为“指控抹黑类线索”。

但不建议直接把这些规则作为最终标签覆盖器，因为规则阈值和关键词如果根据 `val.csv` 反复调整，容易形成验证集过拟合。更稳妥的使用方式是：

1. 作为解释输出中的“证据模式归纳”，不直接改标签。
2. 或者作为辅助特征加入模型，并且只使用 `train_clean.csv` 内部 dev / 5-fold 交叉验证决定是否保留。
3. 如果用于 LLM 复核，也应保持高置信覆盖门槛，并报告新增 false positive 的风险。
